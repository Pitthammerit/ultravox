# v0.19.0 — Per-mode auto-switching by app (Finder picker)

**Status:** SPEC. HELD until other v0.18.x fixes are done. Do not implement
without user go-ahead.

**Decided 2026-05-11** by Benjamin Kurtz after reviewing the v0.18.8
opt-in auto-mode implementation.

## Summary

Replace the global on/off `autoModeEnabled` toggle (shipped in v0.18.8)
with a **per-mode app list**. Each `VoiceMode` declares which apps it
auto-activates for, populated by the user via a native macOS app
picker. The static curated `apps.json` registry becomes a one-time
**seed** of suggested defaults — not the source of truth.

## Why this is a better design

The v0.18.8 implementation is the right MVP — a single opt-in toggle
restoring the pre-v0.11.7 behavior. But it has three structural
limitations the user wants to address before promoting auto-mode to
"on by default":

1. **Users can't customize the mapping.** The curated `apps.json`
   ships with ~20 bundle-ID → mode entries. If a user has Mimestream
   (not in the registry), auto-mode does nothing for their mail
   workflow. They can't add their own apps without editing the
   shipping JSON.
2. **One app → one mode is restrictive.** If the user wants Slack to
   auto-switch to a "casual" mode but Slack-in-incognito-channels to
   use "professional message", the bundle-ID-keyed registry can't
   express that. Per-mode lists put the choice in the right place.
3. **The mental model inverts naturally.** "Which apps should this
   mode kick in for?" is what users actually think about when they're
   building modes — not "which mode does this app prefer?". The
   per-mode shape matches the user's mental model.

## Data model changes

```ts
// VoiceMode interface (apps/ultravox/src/lib/voiceModes.ts):
interface VoiceMode {
  // ... existing fields ...

  /** Apps that trigger this mode when frontmost AND `autoModeEnabled`
   *  is true. Empty array (or omitted) = mode is never auto-selected.
   *  Bundle IDs are case-insensitive on the lookup side. */
  autoModeApps?: Array<{
    bundleId: string;
    /** Display name surfaced in the picker UI. Cached at pick time so
     *  we don't have to round-trip to AppKit on every render. */
    displayName: string;
    /** Optional path to the .app icon for the picker UI's left rail.
     *  Resolved via NSWorkspace.icon(forFile:) at pick time and stored
     *  as a base64 PNG (~5-10 KB). Skip if cost is too high. */
    iconBase64?: string;
  }>;
}
```

Settings-level `autoModeEnabled` from v0.18.8 stays — it remains the
global on/off switch. When OFF, per-mode lists are ignored entirely.
When ON, the **first mode whose `autoModeApps` includes the frontmost
bundle ID** wins. Ties broken by mode order in `settings.modes`
(user-controlled via drag handle). If no mode matches, fall back to
`settings.activeModeId` — same fallback semantics as today.

## UX: per-mode picker

Inside the Mode editor (`ModesPanel.tsx`), add a new
`AutoModeAppsSection` between the existing mode fields:

```
┌─────────────────────────────────────────────────────────┐
│ AUTO-MODE FOR THIS MODE                               ?│
│                                                         │
│ When you focus one of these apps, recording automati-   │
│ cally switches into "Email" mode. Only takes effect     │
│ when Configuration → Mode Selection → Auto-mode is on.  │
│                                                         │
│ ┌─────────────────────────────────────────────────────┐ │
│ │ 📧 Mail                                           ✕ │ │
│ │ 📨 Outlook                                        ✕ │ │
│ │ 📩 Mimestream                                     ✕ │ │
│ │                                                     │ │
│ │ + Add app…                                          │ │
│ └─────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────┘
```

Click **+ Add app…** → opens a native macOS picker. Three picker
strategies, easiest first:

### Strategy A: AppleScript `choose application`

```applescript
choose application with prompt "Pick an app for Email mode"
```

Pros: zero Rust deps, returns an `application` reference we can query
for `bundleIdentifier` and `displayedName` via further AppleScript.
Cons: prompt styling is dated, no multi-select, slow to launch on
first call (~500 ms).

### Strategy B: `NSOpenPanel` filtered to `.app` bundles

```rust
let panel = NSOpenPanel::openPanel();
panel.setAllowsMultipleSelection_(true);
panel.setAllowedFileTypes_(["app"]);
panel.setDirectoryURL_(NSURL::URLWithString_("/Applications"));
if panel.runModal() == NSModalResponseOK {
    for url in panel.URLs().iter() {
        let bundle = NSBundle::bundleWithURL_(url);
        // bundle.bundleIdentifier() → String
        // bundle.objectForInfoDictionaryKey_("CFBundleDisplayName") → String
    }
}
```

Pros: native multi-select, native icons, fast, no AppleScript prompt.
Cons: more Rust code, requires `objc2-app-kit` + `objc2-foundation`
imports we may not already have, the user can navigate anywhere on
disk and pick non-/Applications targets (acceptable, even desirable).

### Strategy C: Custom React picker fed by NSWorkspace

List `/Applications/*.app` + `~/Applications/*.app` via Rust, sort by
display name, render in a Tauri-modal picker with search box. Pros:
fully styleable; matches the rest of the Ultravox UI. Cons: most
code; we'd be reinventing `NSOpenPanel` poorly. Skip unless A and B
both prove insufficient.

**Recommendation: try B first.** Falls back to A if `objc2` complexity
balloons. C only if we want a polished in-app experience post-v1.

## Seed migration from v0.18.x

Existing users have:
- The curated `apps.json` ⊆ Bundle ID → mode mapping (shipping data)
- `autoModeEnabled` boolean in their settings

On first launch of v0.19.0:

1. Walk `apps.json`. For each entry, look up the matching mode by
   `preferredMode` ID in the user's `settings.modes`. If found,
   append `{ bundleId, displayName }` to that mode's `autoModeApps`
   (de-duped by bundle ID).
2. Preserve `autoModeEnabled` as-is. Users who had it on get all
   curated defaults automatically and can refine. Users who had it
   off see the same default mode-list but with the per-mode lists
   pre-seeded — so the moment they flip the toggle on, they get the
   v0.18.8 behavior.
3. After migration runs once, set a `settings.autoModeSeeded = true`
   flag so we don't re-add user-deleted entries on every launch.
4. `apps.json` stays in the codebase as the seed only. Future
   contributions to the registry only benefit *new* users — existing
   users with `autoModeSeeded === true` keep their own list.

- per-mode-list: app in NO mode's autoModeApps → returns activeModeId

## Out of scope for v0.19.0

- Per-app **custom mode params** (e.g. "auto-switch to Email mode but
  with a different cleanup template for Slack"). Modes are atomic;
  the registry just selects which mode. Sub-mode parameterization is
  a v1.5+ idea if ever.
- Window-class / URL-pattern matching (browser tab → mode). The
  frontmost-app bundle ID is the only signal in v0.19.0.
- Cloud sync of per-mode app lists across the user's Macs. Local-only
  is the v0.19.0 scope.
